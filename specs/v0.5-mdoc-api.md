> Next: [V0.6 — OAuth and Image Upload](v0.6-oauth-image-upload.md).

## What's New

The companion backend service that powers mdoc's authentication and image hosting is live. mdoc-api is a lightweight TypeScript service (Hono framework) deployed as an AWS Lambda function on NexaEdge infrastructure. It handles two things: OAuth2 authorization (so the CLI never holds Google client secrets) and image upload (so local images can be embedded in Google Docs). This is a separate repository (`jaisonerick/mdoc-api`) but a prerequisite for the CLI's OAuth and local image features.

## Demo

```bash
# OAuth2 flow (called by mdoc CLI, not directly by user)
# 1. CLI opens browser to:
#    https://api.mdoc.dev/auth/start?redirect_uri=http://localhost:9876/callback
# 2. mdoc-api redirects to Google OAuth consent screen
# 3. User grants access
# 4. Google redirects back to mdoc-api with auth code
# 5. mdoc-api exchanges code for tokens using client secrets
# 6. mdoc-api redirects to CLI's local callback with tokens
# 7. CLI stores tokens locally

# Image upload (called by mdoc CLI during push)
curl -X POST https://api.mdoc.dev/images/upload \
  -H "Authorization: Bearer <user-token>" \
  -F "file=@./charts/cpu-usage.png"
# Response: {"url": "https://api.mdoc.dev/images/abc123.png"}
```

## Capabilities

### Added in this version:
- OAuth2 authorization endpoint — initiates Google OAuth flow, holds client secrets securely
- Token exchange endpoint — exchanges authorization code for access/refresh tokens
- Image upload endpoint — receives an image file, stores it, returns a public URL
- Image serving — uploaded images accessible at a public URL for Google Docs embedding
- Deployed on NexaEdge infrastructure (ECS or Lambda)
- Domain configured (e.g., `api.mdoc.dev`)
- HTTPS with valid certificate

### Simplified in this version (improved later):
- No image expiration or cleanup policy — images persist indefinitely
- No rate limiting on upload endpoint — rely on auth to prevent abuse
- No image size limits — accept any file size (revisit post-V1.0)

### Not yet available:
- CLI integration with mdoc-api (V0.6)
- Image caching to avoid re-uploading unchanged images (post-V1.0)

## Definition of Done

- [ ] mdoc-api repository exists at `jaisonerick/mdoc-api`
- [ ] OAuth2 start endpoint redirects to Google OAuth consent screen with correct scopes (Docs, Drive)
- [ ] Token exchange endpoint receives Google auth code and returns access + refresh tokens
- [ ] Client secrets are stored securely in the backend (env vars or secrets manager), never exposed
- [ ] Image upload endpoint accepts multipart file upload with auth
- [ ] Uploaded images are stored and served at a public, stable URL
- [ ] Image URLs are compatible with Google Docs `InsertInlineImage` (publicly accessible, valid image MIME type)
- [ ] Service is deployed and accessible at its domain over HTTPS
- [ ] Health check endpoint responds (e.g., `GET /health`)
- [ ] Service handles concurrent requests without errors
- [ ] API responds within reasonable latency (< 2s for auth endpoints, < 5s for image upload)

## Open Questions

- Deployment target: ECS (always-on) or Lambda (pay-per-use)? Given low expected traffic, Lambda may be more cost-effective.
- Domain: `api.mdoc.dev`? Requires DNS setup on NexaEdge infrastructure.
- Image storage: S3 bucket with public read, or serve through the API with caching?
- Should the API require any form of API key beyond the user's Google token, to prevent unauthorized usage?
- Image retention policy: keep forever, or expire after N days?
