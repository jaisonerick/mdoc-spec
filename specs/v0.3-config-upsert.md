# V0.3: Config and Upsert

> Builds on: [V0.2 — Full Markdown](v0.2-full-markdown.md).
> Next: [V0.4 — Images and Footnotes](v0.4-images-footnotes.md).

## What's New

The document is now customizable and reusable. Users can override any style through YAML config files (home-level and project-level), and per-document via frontmatter. Running `mdoc push` on a file that already has a `doc_id` updates the existing Google Doc instead of creating a new one. The tool writes `doc_id` back to frontmatter on first creation, enabling a seamless edit-push-edit-push workflow.

## Demo

```bash
# Project config in ./mdoc.yaml
cat mdoc.yaml
```
```yaml
styles:
  heading1:
    font_size: 28
    color: "#1a365d"
  normal:
    font_family: "Georgia"
code:
  theme: "dracula"
```

```bash
# First push — creates a new doc, writes doc_id to frontmatter
mdoc push proposal.md
# Output: https://docs.google.com/document/d/1newDoc.../edit

# proposal.md now has doc_id in frontmatter:
head -5 proposal.md
```
```yaml
---
title: "Q1 Proposal"
doc_id: "1newDocAbcXyz"
---
```

```bash
# Edit the Markdown, push again — updates the same doc
mdoc push proposal.md
# Output: Document 'Q1 Proposal' (1newDoc...) will be overwritten. Continue? [y/N]
# User types: y
# Output: https://docs.google.com/document/d/1newDoc.../edit

# Skip confirmation
mdoc push proposal.md --yes
```

## Capabilities

### Added in this version:
- Layered config system: built-in defaults → `~/.config/mdoc/config.yaml` → `./mdoc.yaml` (deep merge)
- All style properties customizable: fonts, sizes, colors, spacing, alignment per element type
- Syntax theme configurable via `code.theme` in config
- Frontmatter parsing: `title`, `doc_id`, `folder`, `account`, style overrides
- Unknown frontmatter fields preserved on writeback
- Upsert: if `doc_id` exists in frontmatter, clear and rewrite the existing Google Doc
- Confirmation prompt before overwriting (shows doc title from Google), skippable with `--yes`
- `doc_id` written back to frontmatter on first creation, preserving all existing content
- Title resolution: frontmatter `title` → first H1 → filename
- Inline code color configurable via `styles.code_inline.color`

### Simplified in this version (improved later):
- No Drive folder placement — doc created in Drive root (V0.4+)
- No multi-account — uses default/only account (V0.6)
- No reference commands (`mdoc styles`, `mdoc config`, etc.) — post-V1.0

### Not yet available:
- Images (V0.4)
- Footnotes (V0.4)
- Drive folder placement (V0.4)
- mdoc-api backend service (V0.5)
- OAuth2 via mdoc-api and multi-account (V0.6)
- Distribution and auto-update (V1.0)

## Definition of Done

- [ ] `~/.config/mdoc/config.yaml` is read and merged over built-in defaults
- [ ] `./mdoc.yaml` is read and merged over the result
- [ ] Frontmatter `styles` overrides are merged as the final layer
- [ ] All style properties from the spec (fonts, sizes, colors, spacing, alignment) are configurable and applied
- [ ] `code.theme` in config selects the chroma syntax theme
- [ ] Frontmatter `title` sets the Google Doc title
- [ ] `doc_id` in frontmatter triggers upsert (clear and rewrite existing doc)
- [ ] Confirmation prompt shows the existing doc's title before overwriting
- [ ] `--yes` flag skips the confirmation prompt
- [ ] `doc_id` is written back to frontmatter on new doc creation
- [ ] Frontmatter writeback preserves all existing fields, comments, and content
- [ ] Unknown frontmatter fields are preserved (not stripped)
- [ ] A team sharing `./mdoc.yaml` in their repo gets consistent formatting across members
- [ ] Config with missing/partial values falls back to defaults (never zero/empty)
